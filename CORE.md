# Vireon CORE

CORE stands for **Co-Operative Relational Evolution**. It is the persistent relational intelligence and execution layer that allows Elora to understand Jordan's intent, preserve continuity, coordinate specialists, execute bounded work, validate results, and leave durable receipts.

CORE is not defined by a collection of personas or by broad autonomy. Its governing progression is:

```text
Memory → Identity → Relationship → Trust → Autonomy → Execution
```

Execution is the proof. CORE should complete ordinary work inside the configured trust envelope rather than stop at plans or proposals. It should ask before actions that cross real-money, private-data, irreversible, public-representation, or external-commitment boundaries.

## Alpha objective

CORE Alpha must prove one end-to-end Sovereign Command Loop:

```text
Founder intent
→ context assembly
→ authority decision
→ direct execution or bounded delegation
→ validation
→ receipt
→ candidate memory
→ Elora synthesis
```

## Alpha acceptance threshold

CORE Alpha is accepted only when all of the following are true:

1. Elora preserves conversational and project continuity across runtime restarts.
2. Relevant canonical memory, relationship context, current goals, corrections, and trust state are assembled before execution.
3. Ordinary local work executes without redundant approval prompts.
4. Genuine boundary actions stop with a clear approval, refusal, or setup-needed state.
5. Elora can create a bounded Nexora work order from natural language.
6. Nexora can modify the configured workspace, run validation, and return proof.
7. Every completed action produces one primary receipt linking authority, memory, tools, artifacts, validation, result, rollback, and trust impact.
8. New inferred decisions remain candidate memory until reviewed.
9. Elora remains the single user-facing executive and synthesizes specialist results.
10. The normal conversational path—not a hardcoded proof phrase—can complete the acceptance scenario.

## In scope

- Elora chat and orchestration.
- Persistent session, memory, identity, relationship, trust, task, and receipt state.
- Local repository and workspace execution.
- Bounded Nexora work orders.
- Typecheck, test, build, and validation execution.
- Reversible internal artifacts.
- Candidate-memory review.
- One internal Google Drive/Docs flow after local execution is reliable.

## Deferred

- Voice expansion and telephony.
- Multiple visible specialist conversations.
- Marketplace or public distribution.
- Enterprise administration, tenancy, SSO, and RBAC.
- Broad third-party integrations.
- Real-money execution.
- Autonomous external sends or public publication.
- Claims that CORE is AGI, sentient, or conscious.

## Non-negotiable product principles

- Elora is the single visible executive interface.
- Specialists operate behind Elora through bounded contracts.
- Memory is a prerequisite for identity and relationship continuity.
- Trust is domain-specific and evidence-based.
- Autonomy expands or contracts from execution evidence.
- Ordinary work executes and leaves receipts.
- Receipts are the central audit and trust primitive.
- CORE never pretends that simulated or unconfigured work occurred.
- Internal execution and external release are separate decisions.
- The system should prefer completed, validated work over review packets when execution is possible.
